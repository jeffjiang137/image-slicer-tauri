# 开发规范

本文件定义电商设计助手的产品、交互、代码、安全和发布规范。后续新增功能、修复问题或让 AI 辅助开发时，都应优先遵守这些规则。

## 产品原则

- 面向电商真实工作流，而不是做孤立的小工具。
- 每个功能都要围绕图片生产效率、平台适配、合规检查或素材复用展开。
- 操作路径要短，常用动作要明显，批量任务要可停止、可恢复理解、可复制结果。
- 不用营销式空页面。应用打开后应直接进入可工作的生产界面。
- 预览结果要尽量接近最终导出结果，避免用户导出后才发现排版偏差。
- 涉及发布、覆盖、删除、批量替换等不可逆操作时，要给用户明确反馈或确认。

## 视觉与交互规范

- 桌面应用应保持工作台气质：克制、清晰、可扫描、信息密度合理。
- 主界面优先服务三类区域：图片预览、素材列表、参数/任务控制。
- 按钮文案使用明确动词，例如“导入图片”“导出长图”“开始检查”“复制全部”。
- 状态要可见：导入中、生成中、检查中、失败、完成、已停止都要有明确提示。
- 错误提示要告诉用户下一步怎么处理，例如“请先配置 API Key”比“请求失败”更有用。
- 不要用大面积装饰图、无意义渐变背景或过度圆角卡片。
- 控件尺寸要稳定，列表项、工具栏、按钮和画布不能因状态变化产生明显跳动。
- 中文界面优先使用自然中文，不混用不必要的英文术语。

## 前端代码规范

- 前端入口集中在 `src/main.js`，样式集中在 `src/styles.css`，默认风险词库在 `src/riskLexiconPreset.js`。
- 新增状态字段时，要给出清晰默认值，并检查导入、清空、重置、保存配置等路径。
- 涉及图片列表、图层列表、批量任务的逻辑，必须保持 ID 稳定，不用数组下标作为长期身份。
- 预览逻辑和导出参数要保持同源，避免“看起来是 A，导出是 B”。
- 大任务要有运行状态锁，防止用户重复点击产生并发任务。
- 批量请求必须支持停止或跳过失败项，不能因为一张图失败导致全部结果丢失。
- 本地存储键名要带项目语义和版本后缀，避免未来结构升级冲突。
- 不要把真实 API Key、Token、Cookie 写进前端常量、默认值、测试数据或注释。

## Rust / Tauri 规范

- Rust 后端负责文件系统、图片解码编码、长图拼接、切片导出、OCR、配置保存和外部 API 调用。
- Tauri 命令返回给前端的错误信息要可读，优先用中文描述用户能理解的问题。
- 图片处理要优先考虑大图稳定性，避免一次性制造不必要的超大中间对象。
- 文件路径必须由用户选择、拖入或应用配置产生，不要写死个人电脑路径。
- 配置文件应保存在当前用户应用配置目录，不写入安装目录。
- API Key 保存前必须经过本机用户级保护；读取时只在运行期解密，不写入日志。
- 外部请求要有超时、错误处理和可理解的失败信息。
- Windows 专属能力要用 `cfg(target_os = "windows")` 包裹，避免破坏跨平台编译可能性。

## API 与模型规范

- 应用不得内置真实 API Key。
- 支持多个接口配置时，要区分提示词、风险识别和生图用途。
- 自定义接口应尽量遵循 OpenAI 兼容约定，便于用户接入代理服务或第三方模型。
- 未配置 API Key 时，基础图片拼接、排序、预览、导出功能必须仍可使用。
- 模型调用要展示任务状态；有费用或用量信息时，应尽量展示摘要。
- 提示词模板应可编辑、可保存、可恢复，不把用户习惯锁死在默认模板里。

## 极限词与合规规范

- 极限词排查是辅助工具，不应承诺百分百合规。
- OCR 结果要展示给用户，方便人工复核。
- 风险词库必须支持本地维护，不能只依赖固定内置词表。
- 新增词库分类时，要考虑搜索、恢复默认、导入导出和旧数据兼容。
- 合规提示要谨慎表达，避免写成法律结论。

## 文件与仓库规范

- 必须提交：源码、默认词库、配置示例、README、开发规范。
- 不应提交：`node_modules/`、`dist/`、`src-tauri/target/`、安装包、日志、本地上传数据、`.env`、真实配置文件。
- 新增敏感配置时，同步更新 `.gitignore` 和 README 安全说明。
- 不要提交个人电脑路径、店铺账号密码、Cookie、Token、真实订单或客户资料。
- 提交前检查 `git status`，确认没有误加本地数据目录。

## 构建与发布规范

- 日常开发先运行：

```bash
npm run build
```

- Rust 或 Tauri 后端改动后运行：

```bash
cargo check
```

- 发布 Windows 安装包时使用：

```bash
npm run tauri build -- --bundles nsis
```

- GitHub Release 附件优先使用英文文件名，例如：

```text
ecommerce-designer-1.0.36-x64-setup.exe
```

- Release 说明保持简洁，不暴露 API Key、账号、内网地址或客户信息。
- 普通用户下载入口是 Release 里的 `.exe` 安装包，不是 `Source code`。

## 文档规范

- README 面向使用者和协作者，先说明产品价值，再说明功能和开发方式。
- 功能描述要讲清楚业务场景，不只罗列按钮名称。
- 开发规范写在 `RULES.md`，更新架构或关键流程时同步维护。
- 截图或示例如果包含账号、密码、API Key、店铺后台信息，发布前必须打码。

## 变更边界

- 小功能优先小改动，避免顺手重构无关模块。
- 修改 UI 时要同时检查桌面窗口宽度、列表溢出、按钮文字和空状态。
- 修改导出逻辑时要检查长图导出、切片导出、间距填充和补边是否仍一致。
- 修改 API 配置时要检查旧配置兼容、加密保存、空 Key 状态和错误提示。
- 修改词库结构时要检查默认词库、用户本地词库和恢复默认逻辑。
